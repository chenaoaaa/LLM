## 一、 使用低温度的场景（0.0 - 0.3）
当 Agent 需要精确、可靠、可复现的结果时，使用低温度。

### 代码生成与执行
Agent 编写 Python/Shell 代码、SQL 查询或配置脚本时，必须保证语法正确、逻辑严谨。温度过高会导致代码产生幻觉（如使用不存在的函数名）。

### 结构化数据提取
从非结构化文本中提取 JSON、XML 或特定字段时。需要模型严格按照 Schema 输出，不能自由发挥字段名或格式。

### 工具调用
Agent 使用 Function Calling 时。需要模型精准地选择正确的工具并填入正确的参数，任何随机的参数变更都可能导致工具执行失败。

### 数学计算与逻辑推理
在做数学题、逻辑链推理（Chain of Thought）或需要严格遵循“如果A则B”的规则时。低温度能减少推理过程中的随机跳跃，提高准确率。

### 检索增强生成
当 Agent 基于检索到的知识库回答问题时。需要模型严格忠实于检索到的上下文，避免在事实基础上进行过度“创作”导致幻觉（Hallucination）。

## 二、 使用高温度的场景（0.7 - 1.2 甚至更高）
当 Agent 需要多样性、创意、头脑风暴或探索时，使用高温度。

### 创意写作与文案生成
需要 Agent 撰写营销文案、故事、诗歌或润色文本时。高温度能让词汇选择更丰富，句式更多变，避免输出显得“机械”或“模板化”。

### 头脑风暴与方案探索
让 Agent 生成多种不同的解决方案、列出可能性列表或进行“发散性思维”时。高温度能让 Agent 跳出固定的思维框架，提出意想不到的选项。

### 开放式对话与角色扮演
当 Agent 扮演客服、游戏 NPC 或进行闲聊时。适中的高温度（如 0.8）能让回复更自然、更有“人味儿”，避免每次回复都像复读机一样刻板。

### 复杂任务的自我纠错
在一些高级 Agent 架构中，当模型陷入“死循环”或“确定性的错误答案”时，临时提高温度可以让它跳出局部最优解，尝试新的路径来解决问题。

## 三、 实践中的组合策略
在实际开发 Agent 时，不建议对整个过程使用同一个温度。更高级的做法是动态调整：

ReAct 模式中的区分：

Thought：可以设置稍高的温度（0.5-0.7），让推理过程更灵活。

Action：在输出工具调用时，通过逻辑强行限制温度趋近于 0，确保调用准确。

多 Agent 协作：

执行者 Agent：低温度（0.0-0.2），负责写代码、调 API，必须精确。

规划者/监督者 Agent：中高温度（0.4-0.8），负责拆解任务、评价结果，需要一定的灵活性。

## 总结
低温度 = 工程师模式：追求确定性、可控性、逻辑严密。适合写代码、查数据、调工具、做数学题。

高温度 = 艺术家/探索者模式：追求多样性、创造性、自然度。适合写文案、头脑风暴、闲聊、尝试新路径。

一个参考基线：如果你拿不准，对于大多数偏向“干活”的 Agent，可以从 0.1 或 0.2 开始测试；对于偏向“聊天”的 Agent，可以从 0.7 开始测试。
