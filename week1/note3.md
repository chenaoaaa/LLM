# 1. System（系统角色）
## 作用：设定对话的“底层规则”

System 消息用于定义模型的行为边界、角色身份、输出格式、语气风格等全局性指令。它相当于给模型设定的“人设”或“操作手册”，贯穿整个对话始终。

## 特点：

通常放在消息列表的最开头

在大多数 API 实现中，System 消息的权重较高，模型会优先遵循

不会被模型“记住”为用户输入的一部分，而是作为背景约束

## 典型示例：

text
你是一个专业的客服助手，名叫小智。
你的职责是解答用户关于产品的疑问。
回答时使用亲切友好的语气，不超过 100 字。
如果用户询问超出范围的问题，礼貌地表示无法回答。
使用场景：

设定角色（医生、律师、程序员、客服）

规定输出格式（JSON、Markdown、纯文本）

限制行为边界（禁止回答敏感话题）

设定语气风格（专业、幽默、简洁）

# 2. User（用户角色）
## 作用：代表“人类用户”的输入

User 消息用于模拟用户发出的指令、问题或反馈。在多轮对话中，每一条 User 消息都代表外部输入（可能是真实用户，也可能是系统自动生成的触发）。

## 特点：

代表当前轮的输入内容

可以包含问题、指令、上下文信息、甚至示例

在对话历史中，User 和 Assistant 通常交替出现

## 典型示例：

text
“我上个月买的手机屏幕碎了，能保修吗？”
使用场景：

用户提问

用户下达指令

用户提供补充信息

在少样本学习（Few-shot）中作为示例的输入部分

# 3. Assistant（助手角色）
## 作用：代表“模型”的输出

Assistant 消息用于存放模型生成的回复。在 API 调用中，模型输出的内容会自动填充到 Assistant 角色中。当开发者手动构造历史对话时，也需要将之前的模型回复以 Assistant 角色的形式传入。

## 特点：

通常是 API 返回的内容

在多轮对话中，历史中的 Assistant 消息会成为后续对话的上下文

可以包含模型生成的文本、思考过程（如 ReAct 中的 Thought）、工具调用请求等

## 典型示例：

text
“非常抱歉听到您的手机出现问题。请问您的手机是在官方渠道购买的吗？如果是在官方渠道购买且在保修期内，屏幕碎裂属于意外损坏，需要支付一定的维修费用。需要我帮您查询附近的授权维修点吗？”
使用场景：

存储模型的回答

构建多轮对话历史

在 Agent 架构中，Assistant 消息可能包含工具调用的特殊格式
